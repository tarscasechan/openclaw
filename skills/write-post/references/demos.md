# Demos

Use demonstrations only when they clarify the article's core idea.

## Prefer
- Mermaid architecture graphs for systems posts
- small code blocks for setup or usage
- tiny before/after examples
- one sharp visual that carries a point

## Plan before drafting
- Decide the likely demo shape before drafting: none, Mermaid, code, before/after, table, or image.
- The demo plan should name the job the demo would do.
- Planning a demo does not mean the final post must include it.

## Execute after drafting
- Add the demo only when the surrounding prose proves the visual teaches faster.
- If the draft already explains the idea better, skip the demo.
- Do not add a second visual just because the topic is technical.

## Avoid
- decorative images with no job
- too many diagrams
- diagrams that repeat the prose

## Decision check
- Does it clarify one hard idea?
- Would the reader understand or trust the piece less if it were removed?
- Is it smaller than the explanation it replaces?

## Rule

If the demo does not teach faster than prose, leave it out.
